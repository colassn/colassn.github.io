電子手冊 Pro｜Codex 固定開發規則

0. 專案資料

本專案是 GitHub Pages 網站：

- Repository："colassn.github.io"
- 正式版主程式："/rchw/index.html"
- 測試 / 更新版本目錄："/backup/"
- 網站名稱："電子手冊 Pro"
- 類型：大型單檔 HTML / CSS / JavaScript / Firebase App
- AdSense Publisher ID："ca-pub-4156749891284054"

Codex 必須把本專案視為完整正式產品，不可以簡化成 demo，不可以重寫成低配版。

---

1. 最重要規則

1. 不可以直接修改 "/rchw/index.html"。
2. "/rchw/index.html" 是正式版，只能在使用者明確說「發布正式版」時才可以覆蓋。
3. 所有測試 / 更新版本 HTML 都必須放在 "/backup/" 目錄。
4. 不再使用 "/rchw/v1.8.x.html" 作為主要測試版本位置。
5. 每次新版本都必須基於「上一個使用者確認成功的版本」繼續修改。
6. 如果使用者說「成功」、「呢個版本成功」、「可以用」、「正常」、「呢版 OK」、「呢版冇問題」，就把該版本視為下一次更新的基礎版本。
7. 下一次製作時，必須先讀取上一個成功版本，再複製成新的 "/backup/" 版本檔案。
8. 不可以自行退回舊版本。
9. 不可以基於 "/rchw/index.html" 重新開始，除非使用者明確要求。
10. 每次只修改新的 "/backup/" 版本檔案。
11. 不可以刪除舊版本檔案。
12. 不可以刪除正式版。

---

正式版：

測試 / 更新版本必須放在：

版本檔案例子：

/backup/v1.8.46.html
/backup/v1.8.47.html
/backup/v1.8.48.html

測試網址格式：

https://colassn.github.io/backup/v1.8.46.html

1. 本次基於哪個成功版本。
2. 本次會建立哪個新版本檔案。
3. 本次會修改哪些區域。
4. 是否會修改 "/rchw/index.html"。
5. 如何確保只修改新版本檔案。

如果使用者沒有明確指出上一個成功版本，Codex 必須先回報自己找到的候選版本，等待使用者確認，不可以自行亂揀。

3. 預設模式：日常安全更新模式

除非使用者明確說「進入大型改版模式」，否則一律使用日常安全更新模式。

日常安全更新模式下：

1. 基於上一個成功的 "/backup/" 版本。
2. 建立新的 "/backup/" 版本檔案。
3. 只修改新版本檔案。
4. 不修改 "/rchw/index.html"。
5. 不大幅重構整個專案。
6. 不修改與本次需求無關的頁面。
7. 不刪除功能。
8. 不改動核心登入、Firebase、AdSense、管理後台邏輯。
9. 優先做最小必要修改。
10. 每次改動都要方便回退。

適合任務例子：

設定頁面排版太醜，幫我修改去靚啲同埋排版正常啲

進階設定按鈕點擊不到，幫我修復

背景顏色太普通，幫我改善到更自然

4. 大型改版模式

只有當使用者明確說出以下其中一句，Codex 才可以進入大型改版模式：

大型重構
大更新
重新設計整個介面
允許大幅修改
允許大改

大型改版模式下允許：

1. 大幅修改 UI 結構。
2. 大幅重整 CSS。
3. 大幅重整 HTML 結構。
4. 重構 JavaScript 組織方式。
5. 重新設計多個頁面。
6. 合理移動功能入口。
7. 合理調整頁面架構。
8. 合理刪除重複、廢棄、沒有被使用的舊 UI 區塊。
9. 將介面改成更像真正手機 App。
10. 將舊式版面重整成更現代的產品設計。

但大型改版模式仍然必須遵守底線：

1. 不可以直接修改 "/rchw/index.html"，除非使用者明確說「發布正式版」。
2. 必須基於上一個成功的 "/backup/" 版本。
3. 必須建立新的 "/backup/" 版本檔案。
4. 不可以刪除核心功能。
5. Google login 必須保留。
6. Email login 必須保留。
7. Firebase 必須保留。
8. AdSense 必須保留。
9. 管理員模式和後台入口必須保留。
10. 個人功課、時間表、科目管理、班級功能必須保留。
11. 必須完成 syntax / parser check。
12. 必須回報修改前後行數和檔案大小。
13. 必須列出可能受影響的功能。
14. 必須提供測試網址。
15. 必須讓使用者可以回退到上一個成功版本。
16. 不可以生成簡化 demo。
17. 不可以因為大改而移除原本核心資料讀寫邏輯。

大型改版模式下，Codex 不需要過度保守，但必須確保：

- 功能完整
- 可測試
- 可回退
- 不直接動正式版
- 不生成簡化 demo

Codex 不可以：

1. 將完整程式改成 demo。
2. 將大型完整檔案改成簡化版。
3. 刪除 Firebase 設定。
4. 刪除 Google login。
5. 刪除 Email login。
6. 刪除 AdSense。
7. 刪除管理後台入口。
8. 未經要求修改登入邏輯。
9. 未經要求修改資料庫結構。
10. 未經要求修改正式版 "/rchw/index.html"。

12. 只改 CSS 名稱但沒有真正改善畫面。
13. 改錯相似區域但沒有改到真正問題位置。
14. 使用太重動畫導致卡頓。
15. 使用過度浮誇、明顯 AI 生成感的設計。

7. 禁止使用原生 UI 控件

不要新增或使用以下原生控件：

<select>
<option>
<input type="checkbox">

如需要相關功能，請改用：

- custom dropdown
- custom switch
- segmented control
- custom modal
- custom confirmation dialog
- custom color picker
- custom picker sheet
- custom action sheet

所有 UI 要像真正手機 App，不要像普通瀏覽器表單。

---

8. 禁用文字

介面文字和新增程式碼中不要出現以下詞語：

- 同步
- 手動同步
- 手動加入
- 等待加入
- 加入個人清單

如原本功能需要表達類似意思，請改用其他自然講法。

9. UI 設計方向

整體設計方向：

- Apple / Linear / Notion 風格
- 現代
- 乾淨
- 高級
- 克制
- 真正手機 App 感
- 不浮誇
- 不要明顯 AI 生成感
- 低飽和度顏色
- 清楚留白
- 排版自然
- 點擊有即時回饋
- 頁面切換順滑
- 動畫輕量
- 不要卡頓
- 支援 reduced-motion
- 深色 / 淺色模式都要正常
- 手機版優先
- iPad 版不能太空或太闊

10. 背景與主題

1. 背景可以根據使用者在設定中選擇的主題色產生細微變化。
2. 背景要是柔和、有層次、有質感的流動漸變。
3. 不要純色背景。
4. 不要太花。
5. 不要太光。
6. 不要影響文字可讀性。
7. 深色模式和淺色模式都要好看。
8. 低效能手機上都要流暢。
9. 動畫要克制。
10. reduced-motion 開啟時要減少動畫。

2. 點擊事件 function。
3. 相關 CSS class。
4. 是否有 mobile / desktop 兩套 UI。
5. 是否有重複 UI。

9. 是否有資料狀態依賴。
10. 準備修改的精確位置。

如果未定位清楚，不要直接修改。

16. 工作流程

每次任務必須按以下流程：

第一步：只分析，不修改

先讀取：

1. "AGENTS.md"
2. 上一個使用者確認成功的 "/backup/" 版本
3. 如需要，讀取 "/rchw/index.html" 作參考，但不修改

然後回報：

1. 本次基於哪個成功版本。
2. 本次要建立哪個新版本。
3. 找到的相關 HTML / JS / CSS 位置。
4. 會修改哪些區域。
5. 如何確保不修改正式版。
6. 會做哪些檢查。

第二步：等使用者確認

除非使用者明確要求直接修改，否則第一步分析後要等待使用者確認。

第三步：建立新版本檔案

從上一個成功版本複製成新的 "/backup/" HTML 檔案。

第四步：只修改新版本

所有修改只允許寫入新的 "/backup/" 版本檔案。

第五步：檢查

完成後必須做完整檢查。

第六步：回報

清楚回報改了甚麼、怎樣測試、是否正常。

19. 使用者簡短指令理解

如果使用者說：

設定頁面排版太醜，幫我修改去靚啲同埋排版正常啲

Codex 應該理解為：

1. 使用日常安全更新模式。
2. 基於上一個成功 "/backup/" 版本。
3. 建立下一個新 "/backup/" 版本。
4. 只修改新版本。
5. 優化設定頁排版。
6. 保留所有功能。
7. 不修改正式版。
8. 完成後提供檢查結果。

如果使用者說：

進入大型改版模式，重新設計整個設定頁

Codex 應該理解為：

1. 使用大型改版模式。
2. 允許較大幅度修改 UI 結構。
3. 仍然基於上一個成功 "/backup/" 版本。
4. 仍然建立新 "/backup/" 版本。
5. 仍然不可直接修改正式版。
6. 仍然必須保留核心功能。

如果使用者說：

呢個版本成功

Codex 應該理解為：

1. 該版本成為下一次更新基礎。
2. 下一次不應退回舊版或正式版。
3. 下一次要從這個成功版本複製出新版本。

20. 最終原則

穩定性比大改重要。
完整功能比視覺效果重要。
成功版本繼承比重新開始重要。
只改新版本比直接改正式版重要。
先定位再修改比盲改重要。
大型改版可以放寬結構限制，但不可以放棄核心功能。
每次都要可回退、可測試、可確認。

Codex 必須嚴格遵守本檔案。
